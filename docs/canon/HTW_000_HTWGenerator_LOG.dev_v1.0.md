# HTW Generator — Dev Log v1.0

**Role:** short development spine  
**Purpose:** explain how the current direction emerged without turning into a full retrospective

---

## 1. Project lineage

HTW Generator began as a three-tab concept:

- Icon
- Effects
- Bitmap

That skeleton remained valid across the project and is still the correct family structure.

---

## 2. Strong modular milestone

A key milestone in the project was **0.6.0a modular**.

Why it matters:
- it became the strongest trusted modular reference
- it protected the project from collapsing back into a monolith
- it locked several important product truths around Icon and shared system structure

0.6.0a is historically important even though current repo truth has moved forward.

---

## 3. Drift phase

After the strong modular base, the project went through a drift-heavy phase.

Symptoms:
- multiple fixes around Effects
- overuse of donor references
- conceptual blur between rails / checker / noise / layers
- chat-based continuity scattering
- implementation pressure under time pressure

This did not destroy the project, but it raised continuity cost.

---

## 4. Concept recovery

The project later recovered several crucial truths:

- historical files were reclassified as **donor references only**
- product truths were re-closed through Chapter 1 / 2 / 3
- the shell still made sense
- modularity stayed alive
- the product family remained intact

This was a conceptual recovery, not yet a full documentation cleanup.

---

## 5. Current repo-era reading

The repo continued to move through later versions up to **0.6.5k**.

That means the project now has two important temporal anchors:

- **0.6.0a** = strongest trusted modular historical base
- **0.6.5k** = current repo truth to audit

One of the main reasons this doc exists is to stop future handoffs from confusing those two roles.

---

## 6. Why this doc pack exists now

This pack exists because the project is now in a better position to work through Claude CLI, but only if:

- product truth is compressed cleanly
- technical truth is separated cleanly
- continuity is separated cleanly
- the repo is audited as current reality

In other words:
the project no longer mainly needs more speculative ideation.
It needs clearer alignment surfaces.

---

## 7. Current development reading

Today the correct move is:

1. audit 0.6.5k against the locked product docs
2. phase the roadmap
3. only then return to direct implementation

That is the cleanest continuation of the project’s actual evolution.
